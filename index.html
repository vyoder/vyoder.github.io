Why use Graph DB over Relational Analytics?¶

Graph databases understand the connections in large and varied data sets

  • Ease Visualizatio and eben help Finding Trilogies
  • Analysis of graph structured data help usto provide invaluable insights about the relationships between entities, as well as enable optimizations over a network of interconnected objects [w.r.t. Conventional Methods]
  • Very simple compared to RDBMS, and much more flexible
  • The real power is in relationship-focused data (most NoSQL dbs don't treat relationships as first-order)
  • As related-ness and amount of data increases, so does advantage of Graph DBs
  • Much closer to our whiteboard model
  • Calculate best route by certain algorithms [DIJKSHTRA's and other graph Algo] of connected semi structured data
  • Ability to adapt changes better than most.
  • 1000 times faster than MySQL Queries

Section 5. Problem Structure [Breaking it down step by step]¶

[More Formally]¶

Given:¶

  • Athenas Owl's MetaData Jsons
  • A set of entities $A,B,C and D$

Problem/Issue :¶

We need to explore the naturally occurring structure of AO metadata to come up with an appropriate ontological structure (e.g. creating communities). Using this structure, we should be able to identify patterns in the data that can help us solve use-cases like

  1. Searching through graphs faster (doing better on time and space complexity)
  2. Searching for similar or frequently occurring things and recommending them (e.g. finding community partners)
  3. Mining graphs to detect outliers on new data (validating community partners)

Task :¶

Predict the correct We should identify all possible use-cases (in addition to the ones mentioned above) and come up with an appropriate structure for our content based taxonomy. Once we have identified the structure, we should implement a prototype of an end to end pipeline which can do the following

  1. Ingest raw content and process it to create the relevant input data for creating a Graph Structure.
  2. Create a taxonomy out of it using some API / Platform / Library / Open source framework / Architecture.
  3. Save the data to a database or in an appropriate file-format
  4. Add data to that taxonomy (structure should update accordingly) to make it more robust across shows / genres
  5. Delete data from the taxonomy (structure should update accordingly)
  6. Perform Common Data Operations like
    • Visualising it for the different use-cases
    • Extracting information which can help in addressing the different use-cases
